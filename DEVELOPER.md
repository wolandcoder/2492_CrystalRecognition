# Документация разработчика — система автоматического захвата кристаллов

> Проект для АО «Болховский завод полупроводниковых приборов» (БЗПП).
> Программно-аппаратный комплекс распознавания, позиционирования и захвата
> полупроводниковых кристаллов вакуумной трубкой манипулятора под управлением
> Mach3.

Этот документ — мой технический рассказ о том, как устроена программа изнутри:
какие модули за что отвечают, какие классы я завёл и почему именно так,
как организован поток данных от камеры до G-кода и какие нюансы стоит знать,
прежде чем что-либо править.

---

## 1. Зачем это всё

На производстве стоит установка с камерой над рабочим полем и манипулятором
с вакуумной трубкой, которым управляет Mach3. Кристаллы (мелкие
полупроводниковые элементы, размер около 0.3 мм) лежат хаотично, и нужно:

1. увидеть их в кадре;
2. понять, какой из них — «верхушка» (нужная сторона), а какой — низ;
3. выбрать ближайший к центру;
4. перевести его положение из пикселей в миллиметры мирового пространства;
5. учесть смещение вакуумной трубки относительно камеры;
6. дать Mach3 команду подъехать туда и захватить.

Всё это работает в реальном времени, поверх живого видеопотока, с GUI на Flet,
горячими клавишами, ручным джогом по осям, паузой, захватом кадра и
калибровкой масштаба. Отдельный CLI (`main.py`) есть для быстрых проверок без
GUI и для администрирования конфига.

## 2. Стек и зависимости

```text
Python 3.10+
opencv-python  >= 4.9, < 5.0   — захват видео, обработка, морфология, watershed
numpy          >= 1.26, < 3.0  — массивы и линейная алгебра
flet           >= 0.25         — GUI (десктоп через Flutter), асинхронный
```

Зависимости лежат в `requirements.txt`. Узких мест по версиям я старался не
вводить — главное, чтобы OpenCV был четвёртой ветки, иначе ломается часть
вызовов морфологии и watershed.

Дополнительно в репозитории есть:

- `node_modules/` и `package.json` с `tsx` — это вспомогательный девинструмент,
  на основной рантайм он не влияет;
- `__core/__movement/moch3_lib/` — встроенная библиотека для общения с Mach3
  (со своим `README.md`).

## 4. Архитектура: как это всё стыкуется

Если совсем коротко, картинка такая:

```text
   ┌────────┐       ┌──────────────┐       ┌────────────────┐
   │ Камера ├──BGR──► VideoSource  ├──кадр─► NeuralManager  │
   └────────┘       └──────────────┘       │ (pipeline      │
                                           │  нейро-модов)  │
                                           └──────┬─────────┘
                                                  │ shared{}
                                                  ▼
                                          ┌──────────────┐
                                          │ _StreamEngine│ (gui.py)
                                          └──────┬───────┘
                                                 │ JPEG base64
                                                 ▼
                                          ┌──────────────┐
                                          │   Flet UI    │
                                          └──────┬───────┘
                                                 │ команды оператора
                                                 ▼
                                         ┌───────────────┐
                                         │ Mach3Bridge   │
                                         └──────┬────────┘
                                                │ G-code (TCP)
                                                ▼
                                         ┌───────────────┐
                                         │ mach3_server  │
                                         │  + Macropump  │
                                         └──────┬────────┘
                                                │
                                                ▼
                                          Манипулятор
```

Сводные диаграммы более красиво расписаны в `ARCHITECTURE.md` (mermaid),
там же — последовательность «GUI → оркестратор → зрение → mach3».

Ключевые принципы, которые я держал в голове:

- **Чистый слой данных**. Конфиги — это `@dataclass`, никакого «гуляющего»
  словаря по коду.
- **Pipeline нейро-модов**. Каждый «мод» — это самостоятельный класс с одним
  методом `apply(frame, context)`. Они выстраиваются в цепочку и общаются
  через `context.shared` (обычный словарь).
- **GUI ничего не считает сам**. Flet-страница только отображает кадр и
  команды; вся логика обработки — в нейро-модах, всё про манипулятор — в
  `Mach3Bridge`.
- **CLI и GUI — два равноправных входа** в одну и ту же систему.
  Они оба используют `ConfigService`, `VideoSourceFactory`, `NeuralManager`.

## 5. Конфигурация (`config.json` + `ConfigService`)

### 5.1 Что лежит в `config.json`

Файл единый и хранит сразу несколько секций:

```json
{
  "app":              { "version", "window_mode", "crystal_size_mm", "video_scales" },
  "source":           { "type", "camera": {...}, "file": {...} },
  "window":           { "title" },
  "neural":           { "mods": [...] },
  "particle_centers": { "color_top", "color_bottom", "color_center", "color_hold" },
  "particle_nearest": { "color_highlight", "color_line", ... },
  "particle_grid":    { "scale_x", "scale_y", "unit", "origin_offset_*", "invert_y", ... },
  "particle_pin":     { "color_pin", "color_trail", "trail_length", ... },
  "movement":         { "mach3_ip", "mach3_port", "vacuum_offset_x/y", "feed_speed*" }
}
```

Я сознательно сделал «толстый» конфиг: все настройки в одном месте, и когда
нужно настроить установку, оператор/инженер открывает один файл, а не лезет в
пять.

### 5.2 Кто что читает

- `ConfigService` (`__core/__camera/config_service.py`) — официальный
  загрузчик/сохранитель. Он работает только с тремя секциями:
  `source`, `window`, `neural`, потому что это «ядро» источника видео и
  пайплайна.
- Каждый мод сам читает свою секцию (`particle_*`) при первом вызове `apply()`.
  Это компромисс: моды получаются полностью самодостаточными, но цена в том,
  что у нас две точки чтения конфига. Я решил, что это допустимо, потому что
  моды — фактически плагины: их можно подключать/отключать, и тащить за собой
  расширение `ConfigService` каждый раз — лишний оверхед.
- `gui.py` дополнительно читает `app` и `movement` напрямую через
  `json.load`, потому что это GUI-специфика (масштаб видео, IP манипулятора) и она
  меняется чаще.

### 5.3 Модели

`config_models.py` целиком на `dataclasses`:

```text
AppConfig
├── source:  SourceConfig
│   ├── type: "camera" | "file"
│   ├── camera: CameraSourceConfig (index)
│   └── file:   FileSourceConfig   (path, loop)
├── window:  WindowConfig (title)
└── neural:  NeuralConfig (mods: list[str])
```

`AppConfig.from_dict()` парсит сырой словарь, `AppConfig.validate()` гоняет
валидацию (тип источника, существование файла, формат имён модов — должны
начинаться с двойного подчёркивания). На этом же шаге ловятся «битые» конфиги
до того, как мы успеем открыть камеру.

### 5.4 Сохранение

`ConfigService.save()` устроен так, чтобы **не терять чужие секции** при
записи: он сначала читает текущий JSON, обновляет только `source/window/neural`,
а остальное оставляет нетронутым. Это важно, потому что секции модов и
`movement` правит GUI и сами моды, а не `ConfigService`.

## 6. Источник видео (`video_source.py`)

Я завёл иерархию вокруг `cv2.VideoCapture`, чтобы остальной код
не видел разницы между «камерой» и «видеофайлом»:

- `FrameResult` — простой `dataclass` с полями `ok: bool` и `frame`.
- `VideoSource` — абстрактный базовый класс с методами `read()` и
  `release()`.
- `CameraVideoSource` — `cv2.VideoCapture(camera_index)`.
- `FileVideoSource` — `cv2.VideoCapture(path)`. Если включён `loop=True` и
  поток закончился, я переоткрываю файл и продолжаю.
- `VideoSourceFactory.from_config(config)` — единственная фабрика, которой
  пользуются и CLI, и GUI. По `config.source.type` отдаёт нужную реализацию.

Так что добавить, например, RTSP-источник или захват с экрана — это просто
ещё один наследник `VideoSource` плюс одна строка в фабрике. Остальное
работать будет как было.

## 7. Нейро-моды: сердце системы

### 7.1 Базовый протокол (`__neural/base.py`)

Я сделал ультра-минимальный API, чтобы добавить новый мод было быстрее.
Всё крутится вокруг двух классов:

```text
FrameContext
  fps:           float          # текущая частота кадров
  source_type:   str            # "camera" | "file"
  source_label:  str            # индекс или путь
  frame_width:   int
  frame_height:  int
  frame_index:   int            # порядковый номер кадра
  overlay_line:  int            # «свободная» строка для текстовых оверлеев
  shared:        dict[str, Any] # общий словарь между модами

NeuralMod
  name: str = "__base"
  def apply(self, frame, context: FrameContext) -> Any: ...
```

`FrameContext.reserve_text_lines()` я сделал, чтобы моды могли независимо
писать текст в верхнюю часть кадра и не накладываться друг на друга
(каждый «бронирует» себе строки и сдвигает счётчик).

Никаких сигналов, очередей, корутин — кадр прилетает, мод его поправил
in-place, отдал дальше. Это синхронный pipeline. Так понятнее, и это
действительно укладывается в realtime на CPU при разрешении 640×480.

### 7.2 Менеджер (`__neural/manager.py`)

`NeuralManager` решает три задачи:

1. **Загрузить** моды по списку имён из конфига. Импорт — динамический
   через `importlib.import_module(...)`. Каждый мод-модуль обязан
   определить класс `Mod`, который наследует `NeuralMod`. Если этого нет —
   `RuntimeError` сразу же на старте, никаких отложенных взрывов в рантайме.
2. **Прогонять кадр через pipeline** в `apply(frame, context)`. Порядок
   вызова мод == порядок в `config.neural.mods`. Я об этом ещё скажу,
   потому что **порядок имеет значение**: например, `__particle_centers_grid`
   читает результаты `__particle_centers`, а `__particle_centers_movement`
   — результат `__particle_centers_nearest`.
3. **Дать список доступных модов** через `NeuralManager.list_available()` —
   GUI и CLI используют его для отрисовки чекбоксов и валидации `enable`
   команд.

Список доступных модов лежит в `__neural/mods/__init__.py`:

```python
AVAILABLE_MODS = [
    "__hud_info",
    "__particle_centers",
    "__particle_centers_pin",
    "__particle_centers_grid",
    "__particle_centers_nearest",
    "__particle_centers_movement",
]
```

Чтобы добавить новый мод:

1. Создать `__core/__camera/__neural/mods/__my_mod.py`.
2. Определить класс `Mod(NeuralMod)` с уникальным `name` и методом `apply`.
3. Вписать его имя в `AVAILABLE_MODS`.
4. (Опц.) добавить описание в `MOD_META` в `gui.py`, чтобы он красиво показывался в настройках.

### 7.3 Зависимости между модами

Я не стал делать честный граф зависимостей в рантайме (хватает того, что
GUI в окне настроек проверяет deps по `MOD_META`). Считается,
что если `__particle_centers_nearest` стоит впереди `__particle_centers`,
то это ошибка конфигурации, и он просто получит пустой `shared` и ничего
не нарисует.

Сейчас зависимости такие:

- `__hud_info` — независимый.
- `__particle_centers` — обязательный, ничего не требует.
- `__particle_centers_grid` — читает `shared["particle_centers"]`,
  пишет `shared["particle_world"]`.
- `__particle_centers_pin` — читает `shared["particle_centers"]`,
  пишет `shared["particle_pin"]`.
- `__particle_centers_nearest` — читает `shared["particle_centers"]` и
  `shared["particle_pin"]`, пишет `shared["particle_nearest"]`.
- `__particle_centers_movement` — читает `shared["particle_centers"]`,
  `shared["particle_nearest"]` и `shared["particle_pin"]`, пишет
  `shared["particle_movement"]`.

В `gui.py`, в секции `MOD_META`, я держу человекочитаемые описания и список
зависимостей. UI настроек не даст выключить мод, от которого зависят
включённые — он каскадно отключает зависимые.

## 8. Главный мод: `__particle_centers`

Это тот самый «движок» компьютерного зрения. Файл большой, поэтому опишу его поблочно.

### 8.1 Внутренние структуры

- `_Det` — одна детекция на текущем кадре. Поля: `cx, cy, w, h, angle,
  area, score, kind, top_conf`. Использую `__slots__` ради экономии памяти,
  потому что детекций на кадр может быть много (десятки).
- `_Track` — стабильный «трек» между кадрами. У каждого трека уникальный
  `tid` (счётчик класса), сглаживается центр через EMA, угол сглаживается
  отдельно через EMA в пространстве `(sin(2θ), cos(2θ))` — так корректно
  работает с цикличностью 180°. Сторона (`top`/`bottom`) тоже сглаживается
  EMA по `top_conf` с гистерезисом, чтобы не дрожала.

### 8.2 Pipeline кадра (метод `apply`)

```text
1.  resize до WORK_WIDTH = 640                    (детекция стабильнее, FPS выше)
2.  CLAHE + Gaussian blur                          (контраст)
3.  Bright mask:  adaptive + fixed + Canny→fill   (светлые «верхушки»)
4.  Warm mask:    HSV inRange по тёплым тонам      (бронзовые «низы»)
5.  binary = bright OR warm
6.  фоновая модель EMA (BG_ALPHA)                  (давим статичный фон)
7.  открыть/закрыть, эрозия, дилатация             (морфология)
8.  если есть «слипшиеся» большие blob:
        watershed по локальным пикам distance transform
        fallback: split по threshold
        затем k-means по большим остаткам
9.  findContours → _qualify(): площадь, solidity,
        aspect, extent, контраст, дисперсия
10. NMS по расстоянию (NMS_DIST)
11. _update_tracks(): жадное сопоставление треков
        и детекций, EMA-обновление
12. отрисовка: bounding box, оси, цвет по стороне
13. context.shared["particle_centers"] = [...]     (для других модов)
```

Все пороги вынесены в константы класса (`MIN_AREA`, `MAX_ASPECT`,
`MIN_CONTRAST`, `BG_ALPHA`, `MATCH_DIST`, `SMOOTH`, `CONFIRM_HITS`,
`HOLD_FRAMES`, `NMS_DIST` и т. д.). Это сделано специально: мне не хотелось
тащить ещё одну схему конфига для тонких параметров CV — на проде они
почти не меняются. Если когда-то надо будет калибровать — достаточно
вынести их в `particle_centers` секцию `config.json` и распарсить в
`_load_colors` (он уже умеет читать цвета оттуда).

### 8.3 Определение стороны (`_classify_side`)

Это один из самых важных кусков. Метод считает медианные V/S/H в зоне
контура и складывает три скоринга:

- `v_score` — насколько контур «светлый» (V высокий) → top.
- `s_score` — насколько он «не насыщен» (S низкий) → top.
- `warm_pull` — штраф за «тёплый» бронзовый оттенок → bottom.

Итог: `conf = 0.45*v + 0.35*s − 0.30*warm`, отсечка по 0.5 даёт
`kind ∈ {top, bottom}`. В трек идёт «мягкий» `top_conf`, и уже на уровне
`_Track.update` сглаживается с гистерезисом, чтобы сторона не «прыгала»
от шума.

### 8.4 Что отдаёт мод наверх

Список словарей в `shared["particle_centers"]`:

```python
{
  "tid":   int,      # стабильный id трека
  "cx":    int,      # координаты центра в пикселях ОРИГИНАЛЬНОГО кадра
  "cy":    int,
  "w":     int,
  "h":     int,
  "angle": float,    # 0..180
  "side":  "top" | "bottom",
}
```

Дальнейшие моды работают уже только с этим списком — им не надо ничего
знать про CLAHE, watershed и прочую внутреннюю кухню.

## 9. Остальные нейро-моды

### 9.1 `__hud_info`

Самый простой мод, отрисовывает FPS и параметры источника поверх кадра.
Использует `context.reserve_text_lines(2)`, чтобы занять две верхние строки
и не наезжать на других.

### 9.2 `__particle_centers_grid`

Рисует координатную сетку и оси, переводит пиксели в миллиметры.
Здесь важная функция `_px_to_world(px, py)`:

```python
dx = (px - cx_frame) * SCALE_X + ORIGIN_OFFSET_X
dy = (py - cy_frame) * SCALE_Y + ORIGIN_OFFSET_Y
if INVERT_Y: dy = -dy
```

Параметры (`SCALE_X`, `SCALE_Y`, `INVERT_Y`, `ORIGIN_OFFSET_*`, `UNIT`,
`GRID_STEP`, цвета и пр.) лежат в `particle_grid` секции `config.json`.
Сам мод динамически подбирает шаг сетки (`_effective_grid_step`), чтобы
шаг в пикселях оставался в диапазоне 40–200, иначе рисуется «снежинка»
или совсем пустое поле. Внизу справа рисует масштабную «лесенку» с
подписью `1px = X мм`.

Помимо отрисовки, мод раскладывает `shared["particle_centers"]` в
`shared["particle_world"]` — список тех же объектов, но уже в мировых
координатах. Это удобно использовать в дашборде/логе, не пересчитывая
вручную.

### 9.3 `__particle_centers_nearest`

Берёт все объекты с `side == "top"` и выбирает ближайший к центру камеры
по евклидову расстоянию в пикселях. Если активирован пин — возвращает
его, игнорируя «ближайший» алгоритм. Рисует:

- линию от центра кадра до центра выбранного объекта;
- bounding box и крестик главных осей;
- лейбл с XY (мм), расстоянием, направлением (bearing) и tilt-углом.

В `shared["particle_nearest"]` отдаёт компактный словарь:

```python
{
  "tid", "px_x", "px_y", "x", "y", "w", "h",
  "angle", "bearing_deg", "dist", "dist_px", "unit", "side",
}
```

`bearing_deg` рассчитывается с учётом `INVERT_Y`, чтобы соответствовать
системе координат манипулятора, а не «экранной».

### 9.4 `__particle_centers_pin`

Это мод «фиксации цели». Когда оператор кликает «ПИН» в GUI, движок
выставляет `shared["_pin_request"] = tid`. Мод запоминает этот `tid` и:

- ищет именно его в каждом следующем кадре (`_find_by_tid`);
- если потерял — пытается **переподцепить** (`_reacquire`) по близости,
  углу, соотношению сторон и стороне (top/bottom);
- если за `HOLD_FRAMES` так и не нашёл — рисует баннер «PIN LOST»,
  чистит состояние и снимает фиксацию.

Хранит `trail` — историю последних позиций (`deque(maxlen=TRAIL_LENGTH)`)
для рисования «хвоста».

В `shared["particle_pin"]` пишет:

```python
{
  "active":      bool,
  "tid":         int | None,
  "cx", "cy":    int,
  "trail":       list[(x, y)],
  "lost_frames": int,
  "just_lost":   bool,    # одноразовый флаг для UI
}
```

`just_lost` я держу специально: когда ПИН только что окончательно потерян,
GUI должен один раз сбросить состояние кнопки и показать пользователю,
что фиксации больше нет. Без этого флага UI начинал бы дёргать кнопку
каждый кадр.

### 9.5 `__particle_centers_movement`

«Связка» зрения и манипулятора. Берёт ближайший (или зафиксированный) кристалл,
переводит в мир и накладывает смещение вакуумной трубки
(`vacuum_offset_x/y` из секции `movement`):

```python
tube_world_x = world_x + VACUUM_OFFSET_X
tube_world_y = world_y + VACUUM_OFFSET_Y
```

Рисует прицел на месте, куда «приедет» трубка, и ведёт линию от кристалла
до этой точки. В `shared["particle_movement"]` отдаёт:

```python
{
  "crystal_world_x", "crystal_world_y",
  "tube_target_x",   "tube_target_y",
  "tid",
}
```

GUI потом использует именно эти числа в кнопке «→ К кристаллу», когда
оператор хочет послать манипулятор на захват.

## 10. Слой управления манипулятором (`__core/__movement`)

### 10.1 `mach3_lib` — низкоуровневая часть

Это библиотека, которую я не писал с нуля, а интегрировал и подложил под
проект (см. её собственный `README.md`). Архитектура внутри — клиент-сервер
поверх TCP с pickled-сообщениями:

- **Клиент** строит объект `MyData` (содержит `Connection` и `Motion`),
  через `generate_gcode(data)` собирает строку G-кода, через
  `send_command_string(data)` отправляет её на сервер.
- **Сервер** (`mach3_server.py`) слушает порт 5555, пишет команду в
  `Mach3Command.txt`, ждёт окончания движения, читает `DROData.txt` с
  координатами и возвращает их клиенту.
- **Mach3** на стороне манипулятора крутит макрос `Macropump.m1s`, который
  читает `Mach3Command.txt` и применяет G-код, а в `DROData.txt`
  пишет текущие координаты осей.

Подробности по настройке Macropump и серверу — в
`__core/__movement/moch3_lib/README.md`. Я их не дублирую тут, чтобы не
расходились версии.

В `MyData.Motion` намеренно реализованы предохранители:

- `set_z(value)` запрещает выходить за `[0, 37]`;
- `set_x/set_y` могут проверять, что точка входит в круг колбы
  (`ColbaConfig`) при `Z < h`;
- `set_feed` запрещает нулевую и отрицательную подачу.

Это первая линия защиты «дурака» на уровне библиотеки.

Также в `Motion.send_command()` есть несколько именованных макросов:
`tozero`, `fullcollibration`, `allzero`, `zerotocamera`, `zerotozond`.
Они состоят из последовательных шагов и ходят на сервер несколько раз —
это скрипты «обнулиться», «полностью откалиброваться», «съехать на
позицию камеры/зонда». GUI выводит их как кнопки в панели Mach3.

### 10.2 `Mach3Bridge` — высокоуровневая обёртка

`__core/__movement/__init__.py` определяет класс `Mach3Bridge`, который
скрывает работу с `MyData`/`send_command_string` за удобным фасадом:

```python
bridge = Mach3Bridge(ip="192.168.1.125", port=5555)

bridge.connect()                                        # пробуем GET_DRO
bridge.get_position()                                   # → {x,y,z,a}
bridge.move_to(x=10, y=20, feed=700)                    # абсолютно
bridge.move_relative("x", 0.5, feed=700)                # относительно
bridge.send_named_command("allzero")                    # макрос
bridge.move_to_crystal(world_x, world_y,
                       vacuum_offset_x, vacuum_offset_y,
                       feed=700)                        # «прыгнуть к кристаллу»
bridge.send_raw("G92 X0 Y0")                            # сырой G-код
bridge.disconnect()
```

Внутри он держит последние известные координаты (`_last_x/y/z/a`) и
обновляет их после каждого движения через `_refresh_after_move`. Это даёт
GUI быстрый локальный кэш позиции для отрисовки DRO.

Ещё одна тонкость: `connect()` фактически просто делает `get_position()`.
Стейт-машины «подключён/не подключён» в манипуляторе нет, поэтому я считаю
успешный ответ DRO признаком, что «связь есть». Если манипулятор выключен —
бридж останется в состоянии `connected = False`, но программа не упадёт,
просто все методы вернут ошибку. Так задумано: оператору надо иметь
возможность работать с зрением и без манипулятора (например, при настройке
оптики).

`Mach3Bridge.move_to_crystal` — это та самая «бизнес-операция»: к мировым
координатам кристалла прибавляется смещение вакуумной трубки и
двумя `move_on` (X, потом Y) манипулятор прыгает в нужную точку. Я не делаю
тут единой G1-команды, потому что реальная установка лучше отрабатывает
последовательно по осям при таких микро-перемещениях.

## 11. Офлайн-просмотрщик (`viewer.py` и `view_camera.py`)

`CameraViewer` — простой цикл: читает кадр, прогоняет через
`NeuralManager`, показывает через `cv2.imshow`, ждёт `Esc`. Замеряет FPS
по `time.perf_counter()` и собирает `FrameContext` сам.

`view_camera.py` — это оберточка из 15 строк, которая просто грузит
конфиг через `ConfigService`, создаёт `CameraViewer` и запускает.
Я держу её специально: на проде иногда полезно убедиться, что зрение
работает, без поднятия Flet и всех его зависимостей.

`main.py view` делает то же самое, но через CLI-команду. То есть, если
зашёл в shell, можно одной командой запустить «просмотр того, что сейчас
в конфиге».

## 12. CLI (`main.py`)

CLI на `argparse` с подкомандами. Стиль вывода — цветной (через ANSI),
с отключением по `NO_COLOR`/`TERM=dumb`.

Доступные команды:

| Команда                                | Что делает                                                                  |
| -------------------------------------- | --------------------------------------------------------------------------- |
| `python main.py config`                | вывести в stdout текущий `config.json` (только секции `source/window/neural`) |
| `python main.py init`                  | интерактивная настройка источника + модов                                   |
| `python main.py source camera --index 0 [--list]` | выставить камеру с индексом, опц. вывести найденные         |
| `python main.py source file --path /p --loop true` | выставить файл                                              |
| `python main.py list-cameras`          | сканировать камеры (до 10 индексов) и показать разрешение/FPS               |
| `python main.py mods show`             | список включённых модов                                                     |
| `python main.py mods available`        | список доступных модов                                                      |
| `python main.py mods enable --name __X` | добавить мод в конфиг                                                       |
| `python main.py mods disable --name __X` | убрать мод из конфига                                                     |
| `python main.py view`                  | запустить офлайн-просмотрщик                                                 |

В CLI же сидит `_probe_cameras()` — он перебирает индексы 0..9 и для каждого
открывает `cv2.VideoCapture`, проверяет `read()`, считывает разрешение и
FPS. Логика «трёх подряд пустых индексов = стоп» сделана, чтобы не висеть
по 10 секунд на пустой машине.

CLI и GUI пользуются ровно одним и тем же `ConfigService`, так что
изменения через `main.py` сразу подхватятся в `gui.py` после
перезапуска (или при следующем `Старт`).

## 13. GUI (`gui.py`)

Это самый большой файл проекта (~2.4k строк). Я понимаю, что многим он
покажется монолитным — это сделано осознанно. Flet хорошо себя ведёт,
когда страница собирается одной большой функцией с замыканиями: проще
делиться состоянием между обработчиками и не тащить лишних сущностей.
Если GUI нужно будет рефакторить — я сделал так, чтобы это можно было
выделить кусками (panel за panel-ом), потому что внутри функции `main`
разделение по «секциям» уже есть (помечены комментариями
`# ── ... ─────`).

### 13.1 Класс `_StreamEngine`

Это «оркестратор» из `ARCHITECTURE.md`. Он:

- держит ссылку на `VideoSource` и `NeuralManager`;
- умеет `start(config)` / `stop()`;
- метод `read_and_encode()` читает кадр, при необходимости накладывает
  пользовательскую трансформацию (rotate/perspective), прогоняет через
  моды, кодирует в JPEG и возвращает base64;
- хранит последний `shared` от модов (`last_shared`) и последнее JPEG
  (`last_b64`) — это нужно для функции «Захват кадра» и команд манипулятору;
- хранит `pin_tid` (что закреплено), `transform_angles[3]` (углы
  трансформации) и `transform_before_neural` (применять трансформацию
  до или после модов);
- внутри держит свой `Mach3Bridge`, чтобы GUI не пересоздавал его на
  каждый запуск.

FPS считаю как EMA: `fps = 0.9*fps + 0.1*(1/dt)`, чтобы не «прыгало».

### 13.2 Цикл рисования

Я не использую Flet-таймер, потому что мне нужно гонять I/O в потоке и
не блокировать UI:

```python
async def _stream_loop():
    while engine.running:
        b64 = await ev_loop.run_in_executor(None, engine.read_and_encode)
        ...
        video_image.src = f"data:image/jpeg;base64,{b64}"
        page.update()
        await asyncio.sleep(1.0 / TARGET_FPS)
```

То есть тяжёлые шаги (OpenCV, моды) выполняются в дефолтном
`ThreadPoolExecutor`, а на главном корутинном потоке Flet-страницы только
обновление картинки. `TARGET_FPS = 25` — этого хватает для оператора и
не выжирает CPU.

### 13.3 Состояние и UX

Я завёл два «снимка состояния UI»:

- `saved_snapshot` — то, что лежит в `config.json`;
- `running_snapshot` — то, с чем сейчас крутится pipeline.

При любом изменении контролов вызывается `_on_any_change()` — он
сравнивает текущий снимок с `running_snapshot` и подсвечивает кнопку
«Перезапустить», если что-то поменялось «на горячую».

Если оператор жмёт «Старт» с изменениями относительно `saved_snapshot`,
показывается диалог «Сохранить и запустить / Запустить без сохранения /
Отмена». Это специально, чтобы случайно не затирать конфиг, но и не
терять локальные правки.

### 13.4 Панели и кнопки

Кратко по разделам, чтобы понимать, где что искать в коде:

- **Camera/File panel** — сканирование камер, превью, выбор видеофайла,
  чекбокс «зациклить».
- **Mods panel** + `MOD_META` — описания модов с зависимостями. В
  настройках выключение мода каскадно отключает зависимые
  (`_enforce_deps`).
- **Video toolbar** — пауза, захват кадра (`on_capture` рисует
  диалог с увеличительным «стеклом» через cv2.resize), ПИН.
- **Transform panel** — три слайдера X/Y/Z для warp/perspective.
  `_apply_transform` использует `cv2.getRotationMatrix2D` для Z и
  `cv2.getPerspectiveTransform` для X/Y.
- **Machine panel (Mach3)** — состояние подключения, DRO, D-pad для
  джога по X/Y/Z/A с настраиваемым шагом, именованные команды
  (`tozero`, `zerotocamera`, `zerotozond`, `allzero`), кнопка «→ К
  кристаллу» (использует `engine.last_shared["particle_movement"]`).
- **Settings dialog** — все «глобальные» настройки в одном диалоге:
  fullscreen, эталонный размер кристалла, IP/порт манипулятора, смещение
  трубки, скорости, чекбоксы модов.
- **Calibration dialog** — берёт усреднённый размер кристаллов в кадре,
  делит на эталонный размер (мм) и предлагает новый `scale_x = scale_y`.
  Сохраняется в `particle_grid` секции `config.json`.
- **Hotkeys** — `?` показывает диалог-памятку. Биндинги:
  - `Ctrl/⌘+Enter` — старт/стоп;
  - `Пробел` — пауза;
  - `Ctrl/⌘+Shift+C` — захват;
  - `Ctrl/⌘+Shift+P` — пин;
  - `Ctrl/⌘+Shift+T` — трансформация;
  - `Ctrl/⌘+Shift+M` — панель Mach3;
  - `Ctrl/⌘+Shift+X` — перезапуск;
  - стрелки/PgUp/PgDn/Home/End — джог по X/Y/Z/A (если открыт mach3).

### 13.5 Сохранение масштаба видео

`_vid_scale` — это коэффициент, на который домножается размер
`video_frame` (Flet-контейнер). При изменении сохраняется в
`app.video_scales[<WxH>]`. То есть для каждого разрешения видео своё
последнее «удобное» масштабирование. Это маленькая эргономическая
плюшка для оператора.

## 14. Поток данных от камеры до G-кода (полный путь)

Чтобы было совсем понятно, как все компоненты связаны:

```text
1. cv2.VideoCapture (CameraVideoSource / FileVideoSource)
        ↓ FrameResult(ok, frame BGR)
2. _StreamEngine.read_and_encode()
        ↓ (опц. трансформация)
3. NeuralManager.apply(frame, ctx)
   3a. __particle_centers
        ↓ shared["particle_centers"]
   3b. __particle_centers_pin
        ↓ shared["particle_pin"]
   3c. __particle_centers_grid
        ↓ shared["particle_world"]
   3d. __particle_centers_nearest
        ↓ shared["particle_nearest"]
   3e. __particle_centers_movement
        ↓ shared["particle_movement"]
4. _StreamEngine.last_shared = ctx.shared
5. UI (Flet) → пользователь жмёт «→ К кристаллу»
        ↓
6. Mach3Bridge.move_to_crystal(world_x, world_y, vox, voy, feed)
        ↓
7. MyData → generate_gcode → "G1 X.. Y.. F.."
        ↓ (TCP, pickle)
8. mach3_server → Mach3Command.txt
        ↓
9. Macropump в Mach3 → Code(command) → манипулятор едет
        ↓
10. DROData.txt ← обновлённые координаты
        ↓ (ответ обратно по сокету)
11. Mach3Bridge кэширует новую позицию → DRO в GUI
```

## 15. Соглашения и стиль

- **Имена с двойным подчёркиванием** — это не «приватные» в смысле PEP 8,
  а маркер «инфраструктурного слоя» проекта (`__core`, `__camera`,
  `__neural`, `__movement`, моды). Внутри этих пакетов уже обычные
  публичные имена.
- **`from __future__ import annotations`** стоит во всех модулях —
  чтобы можно было свободно использовать `int | None` на Python 3.10
  без оглядки на форвард-референсы.
- **`@dataclass`** для всех «чисто данных» (конфиг, FrameResult,
  FrameContext).
- **`__slots__`** в `_Det`/`_Track` — массовые объекты, экономия памяти
  и скорость атрибутного доступа.
- **Логирование** — стандартный `logging`. В GUI инициализируется в
  начале файла, формат включает таймстамп и уровень. Логгер `mach3bridge`
  пишет от лица модуля движения.
- **Никаких глобальных мутабельных переменных** между модулями. Вся
  межмодовая коммуникация — только через `FrameContext.shared`.

## 16. Как запустить с нуля

```bash
# 1. Виртуальное окружение
python3 -m venv .venv
source .venv/bin/activate            # Windows: .venv\Scripts\activate

python3 main.py mods enable --name __particle_centers_pin
```

3. В GUI открыть «Настройки» → «Манипулятор Mach3», задать IP и порт
   (по умолчанию `192.168.1.125:5555`), смещение вакуумной трубки и
   подачу.

## 17. Калибровка масштаба

Краткий рецепт, чтобы координаты в мм соответствовали реальности:

1. В настройках задать `app.crystal_size_mm` — реальный размер эталонного
   кристалла (по умолчанию 0.3 мм).
2. Запустить поток так, чтобы в кадре было хотя бы 2 кристалла.
3. Нажать «Калибровка масштаба» (иконка «линейка» в шапке).
4. Программа возьмёт средний и медианный размер всех `particle_centers`
   в кадре, посчитает `new_scale = crystal_mm / ref_px` и предложит
   записать в `particle_grid.scale_x = scale_y`.
5. После сохранения GUI автоматически перезапустит pipeline, чтобы
   мод `__particle_centers_grid` подхватил новый масштаб.

Если манипулятор и камера не идеально соосны, дополнительно подправляются
`origin_offset_x/y` (смещение «нуля мира» относительно центра кадра) и
`vacuum_offset_x/y` (смещение трубки).

## 18. Что я бы доделал в первую очередь

Этот раздел — мои честные «todo на потом», чтобы новые разработчики
не тратили время:

1. **Единая схема конфига**. Сейчас секции `particle_*` читаются
   модами напрямую через `json.load`. Логично завести `dataclass` под
   каждую секцию и подгружать через `ConfigService`, а в моды отдавать
   через `FrameContext.shared` ещё до первого `apply`.

   проверяется только в UI. Можно вычислять топологически по

   эталонные ролики; на их основе несложно сделать снапшот-тесты на
   `shared["particle_centers"]`.
